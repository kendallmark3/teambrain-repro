Non-Negotiables

6 Core Principles

Every decision runs through all six. Skip one and you've built on a hidden assumption.

01
🧠

Reasoning Before Solutions

Never jump to an answer. Lay out the thinking first — what's the actual problem, what are the constraints, what does the solution space look like before you pick one path.

In practice: Before proposing a caching strategy, restate why the system is slow and whether caching actually addresses the root cause.
02
📋

Explicit Assumptions

Every assumption must be named. Hidden assumptions are hidden failures waiting for the right stress condition. If you're taking something for granted, say it out loud so it can be challenged.

In practice: "We assume app teams deploy weekly and can absorb a 5-minute maintenance window" — said before designing the migration plan.
03
⚖️

Clear Tradeoffs

Every decision has a cost. Name what you're gaining and what you're giving up. There are no free lunches in system design — the question is whether the tradeoff is worth it for your context.

In practice: "Event-driven gains loose coupling but loses transaction guarantees. We choose events because coupling is our bigger risk."
04
⚠️

Risk Surfaced Early

Flag risks before you build, not after. The cost of surfacing a risk early is a conversation. The cost of discovering it late is a production incident with a post-mortem and an angry SLA review.

In practice: "Risk: if the OTEL collector goes down, buffering strategy is undefined. Define before shipping."
05
✅

No Invented Facts

If you don't know, say so. Fabricated confidence is more dangerous than admitted uncertainty. When requirements are ambiguous, stop and ask — do not fill the gap with plausible-sounding fiction.

In practice: "I don't have enough information about the retention requirements to recommend a storage tier. What are the compliance constraints?"
06
🔍

No Hidden Requirements

Make every constraint visible. Security, compliance, org structure, budget, team capability — if it affects the decision, it belongs in the room. Silent constraints are the most dangerous ones.

In practice: "Unstated constraint: team has no Kubernetes experience. The proposed solution requires it. This changes the recommendation."
$ /intent-check

Run this before every new feature. Answer all 7. If any answer is "I don't know" — stop and ask.

  1. What is the actual problem? — Not what was asked. What problem does it solve?
  2. Who is affected? — Which users, systems, or services does this touch?
  3. What are you assuming? — List at least two. If you can't, think harder.
  4. What are the edge cases? — Empty input, max load, network failure, bad auth?
  5. What could go wrong? — Name the top 3 risks before you start.
  6. What is NOT in scope? — Be explicit about what this change does not do.
  7. Is there a simpler solution? — If yes — why aren't you doing that instead?

Run before every new feature

Forces you to articulate the problem, name assumptions, and flag risks before a single line of code is written. The goal is not to fill out a form — it's to surface what you haven't thought about yet.

1 What is the actual problem?

/review
Run before opening any PR

A structured PR readiness check. Every PR must be able to answer these questions or it's not ready to merge. Prevents "just ship it and see" culture in production systems.

→ What problem does this solve?
→ What assumptions does it make?
→ What did you consider and reject?
→ What are the edge cases and how are they handled?
→ What monitoring or observability does this add?
PRs that cannot answer these are not ready for review.
/arch-review
Run before any architectural change

Triggers a structured architecture review before proposing any new pattern, data model, or service boundary. Ensures changes are justified by intent, not convenience.

→ What is the current state and why is it insufficient?
→ What is the proposed change with explicit tradeoffs?
→ What alternative was considered and why rejected?
→ What breaks or needs migration?
→ Does this change require intent file updates?
Architecture changes without this review are blocked from merge.

The Intent Files

These two files are the foundation. Everything else — code, commands, CI — enforces them.

📄 /intent/architecture.intent.md
  • One sentence: what the system exists to do
  • Primary tradeoff: what you prioritize over what
  • Explicit list of things you do NOT do
  • Named owner and trigger conditions for revisit
If this file is missing — work stops. Resolve intent before building.
📄 /intent/constraints.md
  • Every constraint traces back to declared intent
  • Architectural, deployment, automation constraints
  • Constraint violation protocol (explicit exceptions only)
  • If a constraint can't be justified by intent — remove it
Constraints without justification are noise. Only keep what traces to intent.

The Origin

Why TeamBrain Exists

AI tools got fast. They didn't get better at judgment. TeamBrain was built to close that gap.

The Founding Insight

AI doesn't fail because it's bad at generating code. It fails because it doesn't know what you actually need — and it's too confident to say so.

In enterprise environments, that confidence is dangerous. A junior engineer who says "I'm not sure" is safer than an AI that invents a plausible answer to fill the gap. The cost of fabricated confidence isn't a failing test — it's a production incident, a missed compliance requirement, or six weeks of work built on a misunderstood constraint.

TeamBrain was built from a simple premise: reasoning should happen before code. Not as a formality. Not as documentation. As the actual first step — because the shape of good code depends entirely on the shape of the problem, and the shape of the problem is almost never what it appears to be on the surface.

"The problem is not that AI is wrong. The problem is that AI is confidently wrong in ways that are hard to detect until it's too late."

Core Beliefs

01

Intent before implementation

You cannot write good code for a problem you haven't understood. The intent layer is not overhead — it's the foundation everything else sits on.

02

Constraints are infrastructure

Architecture constraints should be version-controlled, reviewable, and enforced by CI — not carried in people's heads or scattered across Confluence pages nobody reads.

03

AI behavior should be reproducible

If two engineers on the same team get different AI advice for the same decision, the framework has failed. AI operating instructions belong in the repo, not in someone's clipboard.

04

Uncertainty is information

An AI that says "I don't know, clarify this before I proceed" is more valuable than one that fills the gap with a plausible invention. Acknowledged uncertainty is a decision input.

05

Violations must be named, not worked around

If a constraint needs to be broken, that should be an explicit, approved, documented exception — not a silent workaround discovered in a post-mortem. Transparency over convenience.

06

The framework should evolve

Intent is not permanent. When the system's use case changes, the intent file must be revisited. A framework that can't adapt is just technical debt with a nice README.

🌐

Open Source & Community-Driven

TeamBrain is fully open source. The prompts, templates, intent files, CLAUDE.md structure, and CI checks are all available on GitHub. Teams are encouraged to fork, adapt, and contribute back — the framework improves when real teams use it in real environments.

If you find a place where the framework gives bad advice, produces constraint violations, or fails to surface a real risk — open an issue. That feedback is the most valuable input the project can receive.

Real Examples

What TeamBrain actually looks like when applied to real engineering decisions.

Example 1

Running /intent-check Before Adding a Cache

Scenario: A team wants to add Redis caching to a slow API endpoint. Without TeamBrain, they'd start installing Redis. With TeamBrain, they run /intent-check first.

$ /intent-check — Add Redis cache to /api/products
1. What is the actual problem?
The /api/products endpoint takes 800ms average. Users are experiencing slowness during business hours. The problem is response latency — not throughput. We haven't profiled whether the database or the application layer is the bottleneck.
2. Who is affected?
End users on the product listing page. Also the mobile team who call this endpoint on every app launch. Any caching strategy needs to account for both.
3. What are you assuming?
— Assuming the database is the bottleneck (not confirmed by profiling)
— Assuming product data is safe to cache (price changes? inventory?)
— Assuming the team can operate a Redis instance (no current Redis in stack)
— Assuming cache invalidation is straightforward (it never is)
⚠️ Assumption 3 is a hidden requirement: the team has no Redis experience or infrastructure. This changes the recommendation entirely.
4. What are the edge cases?
— Price changes going live but users seeing cached stale prices
— Out-of-stock items showing as available
— Cache stampede under load when cache expires
— Cache miss rate in off-hours vs peak hours
5. What could go wrong?
1. Serving stale product data during a flash sale (revenue / trust impact)
2. Redis becomes a new SPOF — if Redis goes down, does the API fail or fall back?
3. Added ops burden: Redis monitoring, eviction policies, memory sizing — team unprepared
6. What is NOT in scope?
This change does not address the root cause of slowness (may be N+1 queries, missing DB index, or application logic). Caching may mask a fixable underlying problem.
7. Is there a simpler solution?
Yes: add a database index on the most-queried field. 2-hour change, no new infrastructure, no invalidation logic. Should be investigated before Redis is provisioned.
✅ Recommendation: Profile first. Add the DB index. Revisit caching only if latency persists. If caching is still needed, use HTTP cache headers (no new infra) before introducing Redis.
Example 2

Intent Violation Detected During /arch-review

Scenario: A developer proposes direct database sharing between two microservices to "simplify the integration." The AI detects a constraint violation.

$ /arch-review — Direct DB access from Order Service to User Service
⚠️ Intent Violation Detected
Requested action: Order Service reads directly from User Service's database to avoid an API call.
Violated constraint: "Services communicate via contracts (APIs, events). Direct database sharing between services is not permitted." — constraints.md, Architectural Constraints #1
Why it matters: Direct DB sharing creates invisible coupling. If User Service changes its schema, Order Service breaks silently. There is no contract, no versioning, no circuit breaker. Under stress, this becomes a cascading failure path.
Compliant alternative: Expose a GET /users/:id/profile endpoint from User Service with a defined contract. Order Service calls the API. Cache the response at the Order Service layer if latency is a concern. This preserves the service boundary and gives User Service the ability to evolve its schema independently.
If this constraint must be violated (e.g., performance requirements make the API call untenable), follow the Constraint Violation Protocol: name the violation explicitly, get written approval from the intent owner, document the exception with a remediation deadline in constraints.md.
Example 3

Before & After: Migrating to Microservices

Scenario: Leadership asks the team to "break up the monolith." Without TeamBrain, the team starts planning service boundaries. With TeamBrain, they run /arch-review first.

Without TeamBrain
  • 🔴 Team starts drawing service boundaries on a whiteboard
  • 🔴 Nobody asks why — what problem does decomposition solve?
  • 🔴 "Independent deployability" is assumed as a goal
  • 🔴 Distributed transaction complexity is not surfaced
  • 🔴 Team skill gap (Kubernetes, service mesh) goes unstated
  • 🔴 Six months later: a distributed monolith with all the costs and none of the benefits
With TeamBrain /arch-review
  • ✅ Problem restated: "We can't deploy without coordinating 4 teams"
  • ✅ Real goal identified: deployment independence for the payments domain
  • ✅ Alternative considered: modular monolith with clear domain boundaries
  • ✅ Top risks named: data consistency, team skill gap, org structure mismatch
  • ✅ Decision: extract payments only, keep the rest as a modular monolith
  • ✅ Smaller scope, faster delivery, manageable risk surface

# AI System Prompt — Governed Mode
> Stored in the repo so AI behavior is version-controlled
> alongside code. Changes go through PR review.

---

## Paste This Into Your AI Session

You are operating under declared architecture intent
for this project.

The intent file is at: /intent/architecture.intent.md
The constraints file is at: /intent/constraints.md

You must:

1. **Respect defined tradeoffs**
   Do not optimize against declared priorities.

2. **Flag violations**
   If a request contradicts a constraint, say so
   before complying.

3. **Refuse silent shortcuts**
   Never implement a workaround that violates intent
   without naming it.

4. **Explain architectural reasoning**
   When suggesting a change, trace it back to a
   constraint or flag the conflict.

5. **Treat velocity as a tradeoff, not a default**
   Speed is only correct when it does not compromise
   declared priorities.

When you detect a constraint violation, respond with:

⚠️ Intent Violation Detected
Requested action: [what was asked]
Violated constraint: [which constraint this breaks]
Why it matters: [consequence under stress or at scale]
Compliant alternative: [what we can do instead]

You are a coherence engine, not a velocity engine.
Do not generate code until intent is declared.
If intent has not been declared, ask for it.

---

## Why This File Exists in the Repo

Most teams treat AI prompts as ephemeral — pasted into
chat, lost after the session. This file treats AI
operating instructions as infrastructure:

- Version controlled
- Reviewable in PRs
- Traceable to intent
- Consistent across all team members

Any engineer can open a new AI session, reference this
file, and operate inside the same constraints as everyone
else. Changes to this prompt go through PR review and must
reference the intent change that motivated the update.

The framework is AI-agnostic — the principles, intent files, and templates work with any AI. The /intent-check, /review, and /arch-review commands are Claude Code slash commands, but the underlying questions can be run with any tool.
Does this replace agile or sprint planning?
No. TeamBrain works alongside your existing process. It adds a reasoning layer before and after implementation — it doesn't replace how you plan, track, or ship work. Think of it as a decision quality gate, not a delivery methodology.
How long does /intent-check take?
For a well-understood feature: 5–10 minutes. For something ambiguous: however long it takes to stop the ambiguity. The point is not speed — it's preventing the 3-week detour that comes from shipping on a misunderstood requirement.
What if the team skips the commands?
CI checks enforce that intent files exist and are non-empty. PRs that fail intent checks are blocked from merge. The framework is designed to be enforced structurally, not just culturally.
Can we modify the commands for our team?
Yes — the command files live in .claude/commands/ and are version-controlled. Changes go through PR review, and any change must reference the constraint or intent that motivated the update. Prompt drift is treated like code drift.
Who owns the architecture intent file?
A named role — typically the Tech Lead or Architect. That person signs off on changes to the intent file, is accountable when system behavior diverges from declared intent, and leads the intent review when trigger conditions are met.
